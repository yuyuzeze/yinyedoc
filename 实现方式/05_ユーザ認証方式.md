## 5.1 ユーザ認証目標

外部からシステムに対してアクセスを行う際に、利用者の識別を行うことで、システムが所有する情報の保護を実現することを目標とする。

本システムは JAFFIC クラウド上に構築された内部システムであり、信用基金が既に利用中の Microsoft Entra ID テナントを継続使用してシングルサインオン（SSO）認証を採用する。

これにより、職員は既存の Microsoft アカウント認証情報を使用してシームレスにシステムへアクセスでき、新規アカウント作成の必要なく運用を開始できる。

---

## 5.2 認証の構造

MSAL（Microsoft Authentication Library）と Microsoft.Identity.Web を組み合わせて認証アーキテクチャを構築し、各エンドポイントに対してセキュアでステートレスなシステムを作成する。

本システムの認証基盤は信用基金が既に利用中の Entra ID テナントを継続使用する。庁内 Active Directory（AD）は Entra Connect により既に Entra ID と同期済みである。

### MSAL（Microsoft Authentication Library）— Angular フロントエンド

- ブラウザ上の OIDC / OAuth 2.0 認証フローを提供するクライアントライブラリである。
- MsalInterceptor を使用し、全 API リクエストに Access Token を自動付与する。
- ID トークン・アクセストークン・リフレッシュトークンのライフサイクルを管理する。
- トークンはブラウザの sessionStorage に保持し、タブを閉じるまで認証状態を維持する。

### Microsoft.Identity.Web — .net 10 バックエンド

- ASP.NET Core 10 Web API にて Entra ID から発行された JWT アクセストークンを検証する。
- AddAuthentication / AddMicrosoftIdentityWebApi により、トークン名・有効期限・Audience / Issuer を自動検証する。
- 認証成功後、ClaimsPrincipal にユーザ情報（OID・UPN・ロール等）を格納し、[Authorize] 属性によって権限制御を行う。

### Azure RBAC（リソースアクセス権限管理）

- システム単位でグループ化し、RBAC でアクセス許可するユーザおよび権限を管理する。
- ベストプラクティスに則り、付与する権限を最小限にすることで堅密性を確保する。
- Blob Storage・Key Vault・Azure Files 等の Azure リソースへのアクセスは Managed Identity を通じて制御する。

## 5.4 ログイン時の認証

有効なユーザが信用基金の Entra ID テナントを通じてシステムにアクセスできるように保証する。

サーバ側でセッション状態を保存せず、Entra ID が発行した JWT アクセストークンを使用してステートレスな認証を実現する。

### 5.4.1 ログイン時のユーザ認証フロー

> **列構成：** クライアント ｜ APサーバ ｜ Entra ID

```
クライアント              APサーバ                    Entra ID
    │                       │                           │
    │──本システムにアクセス──▶│                           │
    │  （基幹LAN→VPN→Azure）  │                           │
    │                    1.システム初期画面表示            │
    │                    ログイン画面へリダイレクト         │
    │                       │──────────────────────────▶│
    │                       │                MS365サインイン済み？
    │                       │                是 → 自動スキップ（SSO）
    │                       │                否 → ログイン画面表示
    │◀──ログイン画面（否の場合）────────────────────────────│
    │                       │                           │
    │──UPN 入力─────────────────────────────────────────▶│
    │                       │                           │
    │                       │          2.検証を行う（Entra ID が処理）
    │                       │          ・UPN・PW 検証
    │                       │          ・Enterprise Application
    │                       │            グループ割り当てチェック
    │                       │          ・MFA 認証
    │                       │                           │
    │                       │          認証結果          │
    │◀──認証エラー画面（401）──────────────────────────────│（異常）
    │                       │                           │
    │                       │    3.アクセストークン発行    │（正常）
    │                       │    Entra ID → MSAL        │
    │◀──JWT（MSAL が sessionStorage に自動保存）──────────│
    │                       │                           │
    │              4.ロール確認（APサーバ）                │
    │              業務DBのロールテーブルを照会             │
    │              ロールが未割り当て → 403 Forbidden      │
    │◀──ロールなし提示（403）──│                           │
    │                       │                           │
    │              5.認証ログ記録                         │
    │                       │                           │
    │◀──メニュー画面──────────│                           │
    │    6.メニュー画面へ遷移  │                           │
```

**フロー説明：**

**(1) システム初期画面表示**  
クライアントからシステムに対して初回リクエスト要求があった場合、Angular App がロードされ MsalGuard が認証状態を確認する。未認証の場合は Entra ID のログイン画面（Microsoft 標準）へリダイレクトする。基金内 Microsoft 365 等にサインイン済みの場合は自動的にトークンが取得され、パスワード入力画面が表示されない（SSO 動作）。

**(2) 認証を行う**  
Entra ID の認証画面にて UPN（user@organization.com 形式）を入力する。入力された UPN の必須・形式チェックは Entra ID 側で実施する。  
Entra ID が Enterprise Application の設定に基づき特定グループへ属するかを検証する。  
ログイン失敗時のアカウントロック制御は Entra ID のスマートロックアウト機能が自動で管理する。本システム DB にログイン失敗数・ロック状態を持つ必要はない。  
入力された UPN を条件として、Entra ID ディレクトリからユーザ属性・グループ情報・アプリロールを取得する。

**(3) アクセストークン発行**  
認証・チェック処理に合格した後、Entra ID が Angular（MSAL）に対してアクセストークン（JWT）およびリフレッシュトークンを発行する。トークンのライフサイクル管理は Entra ID と MSAL が行う。

**(4) ロール確認**  
認証成功後、業務DBのロールテーブルを照合し、ログインユーザのアプリロールを確認する。ロールが未割り当ての場合は 403 Forbidden を返却する。

**(5) 認証ログ記録**  
ログイン成功後、ログファイルに業務アクセスの記録を書き込む。ログイン失敗・アカウントロックの記録は Entra ID のサインインログに委託するため、本システムでは管理しない。

**(6) メニュー画面へ遷移**  
認証が完了し JWT が MSAL の sessionStorage に保存された状態でメニュー画面へ遷移する。以降のリクエストは MsalInterceptor が自動的に Bearer ヘッダーに JWT を付与する。

## 5.5 ログイン後の認証

ログイン後にシステムが認証を要求する全てのエンドポイントへのユーザのアクセス権限をチェックし、認証するプロセスである。

Authorization ヘッダーの JWT（アクセストークン）の妥当性を検証し、ClaimsPrincipal に記録されたユーザ情報との一致を確認する。

**ログイン済みユーザの全 API リクエストに対して、JWT の正当性とロールの権限を毎回自動検証し、不正アクセスを防止する。**

### 表5-3 .net 10 Middleware Pipeline（リクエスト処理順）

| 順番 | コンポーネント名 | 目的 |
|------|---------------|------|
| 1 | Authorization Request | クライアントから Bearer JWT を含むリクエストを受け取る |
| 2 | ExceptionHandlerMiddleware | 全例外キャッチ |
| 3 | Serilog RequestLogging | リクエスト / レスポンス |
| 4 | UseAuthentication | JwtBearerHandler が Entra ID JWT を検証する |
| 5 | RoleMiddleware | ロールテーブルを照会し、ClaimsPrincipal にロールを追加する（業務開発者が作成）|
| 6 | UseAuthorization | [Authorize] ロール・ポリシー検査 |
| 7 | MapControllers | Controller へルーティング |

### 5.5.1 ログイン後のユーザ認証フロー

> **列構成：** クライアント ｜ APサーバ

```
クライアント              APサーバ
    │                       │
    │──業務画面 API リクエスト▶│
    │  発行                  │
    │                    1.JWT 有効期限チェック
    │                    （MSAL 自動処理）
    │                    ・期限内 → そのまま継続
    │                    ・期限切れ → AcquireTokenSilent()
    │                      成功 → 新 JWT でリクエスト
    │                      失敗 → loginRedirect（再ログイン）
    │◀──認証エラー画面（401）──│（期限切れ・サイレント更新失敗時）
    │                       │
    │                    正常↓
    │                    2.ロールチェック
    │                    （RoleMiddleware + UseAuthorization）
    │                    ・UserRole テーブルを OID で照会
    │                    ・ClaimsPrincipal にロール追加
    │                    ・[Authorize(Roles=...)] でロール確認
    │◀──ロールなし提示（403）──│（ロールなし）
    │                       │
    │                    正常↓
    │                    業務処理実行
    │◀──200 OK──────────────│
```

**フロー説明：**

**(1) JWT トークン検証（.net 10 Middleware）**  
Angular の MsalInterceptor が全 API リクエストの Authorization ヘッダーに Bearer JWT を自動付与する。JWT の有効期限が切れている場合は MsalInterceptor が AcquireTokenSilent でサイレント更新を試みる。更新失敗時は loginRedirect で再ログインを要求する。  
.net 10 側では Microsoft.Identity.Web（JwtBearerHandler）が Entra ID の公開鍵を使って署名・有効期限・Audience・Issuer を自動検証する。

**(2) ロール確認（.net 10 Middleware）**  
RoleMiddleware が JWT の OID をキーに業務DBのロールテーブルを照会し、ロールを ClaimsPrincipal に追加する。その後 UseAuthorization ミドルウェアが JWT の roles クレームから [Authorize(Roles=...)] または [Authorize(Policy=...)] によりロールを確認する。  
ロールが不足している場合は 403 Forbidden を返却する。本処理は .net 10 のミドルウェアが自動で行うため、Controller に個別実装は不要である。

## 5.6 認証履歴管理

認証に関するログ管理は Entra ID（Azure Monitor）と本システムログファイルの2層で構成する。

### 表5-4 ログ管理の責任分担

| ログの種類 | 管理場所 | 内容・備考 |
|---------|---------|----------|
| ログイン成功・失敗 | Entra ID（Azure Monitor）| Azure Portal のサインインログで確認可能。本システム DB への記録は不要である。|
| アカウントロック | Entra ID（Azure Monitor）| スマートロックアウトのログで確認可能。本システム DB への記録は不要である。|
| 業務画面アクセスログ | 本システムログファイル | 誰がいつの画面を操作したかを記録する。|

## 5.7 認証情報管理

ユーザ認証情報のメンテナンス（ロール付与・アカウント有効化等）を行う。

本システムは信用基金の既存 Entra ID テナントを継続使用するため、アカウント管理の責任は以下の通り分担される。

### 表5-5 認証情報管理の責任分担

| 管理項目 | 管理方法 | 管理の責任者 |
|---------|---------|------------|
| アカウント作成 | 基金 IT 管理者（AD 側で作成）| 信用基金 Azure 管理者（Entra ID）|
| パスワード管理 | 基金 AD（Windows パスワード）| Entra ID（Microsoft パスワード）|
| アカウント無効化 | AD 側で無効化 → 自動同期 | Entra ID 側で無効化 |
| アプリロール割り当て | Entra ID（システム管理者）| Entra ID（システム管理者）|
| グループ管理（部署）| 庁内 AD → Entra ID に同期 | Entra ID（Azure 管理者）|

**【権限区分】**

**(1) 管理権限ユーザ**  
Entra ID のシステム管理グループに属するユーザ。管理者メニューからユーザ管理・ロール割り当てを行う。

**(2) 一般ユーザ**  
Entra ID の一般グループに属するユーザ。ユーザが所属している組織に基づくメニューから業務機能にアクセスする。

**(3) ユーザ全般**  
パスワード再設定は Entra ID のセルフサービスパスワード再設定（SSPR）機能、または AD 連携時は基金内 AD のパスワード変更規定に委託する。

**(4) システム**  
バックグラウンドサービス・バッチ処理は Managed Identity またはサービスプリンシパルにより Entra ID 認証を行う。

## 5.8 セッションタイムアウト処理

システムに対して、30 分間の操作がない場合に、MSAL のサイレントトークン更新に失敗した後、認証ログテーブルからセッション情報を削除する。

セッションタイムアウトの 10 分前に、ユーザにタイムアウト延長のダイアログを表示し、「延長」ボタン押下後にセッションタイムアウトを延長する。

MsalService.AcquireTokenSilent によりバックグラウンドでトークンを自動更新する。サイレント更新に失敗した場合は loginRedirect を実行し、再認証フローが起動する。

> ※ アクセストークンの有効期限（既定 1 時間）は信用基金 Entra ID テナントの設定により変更可能

## 5.9 ロールベースアクセス制御

本システムでは、ログイン後のアクセス制御を業務DBのロールテーブルで管理する。権限のない画面はメニューに表示せず、直接アクセスした場合も権限無し画面を表示するのみでロック処理は行わない。

ロール定義は保証担当、管理担当、経理担当、業務推進担当、すべてである。

### 5.9.1 ロールベースアクセス制御フロー

> **列構成：** クライアント ｜ APサーバ ｜ 業務DB

```
クライアント              APサーバ                    業務DB
    │                       │                           │
    │──1.ログイン完了────────▶│                           │
    │  Entra ID の JWT に     │                           │
    │  OID・UPN が含まれる    │                           │
    │                    2.UserRole テーブル照会           │
    │                       │──SELECT WHERE user_oid=───▶│
    │                       │                    UserRole テーブル
    │                       │◀──ロール一覧────────────────│
    │                       │  ClaimsPrincipal に追加     │
    │                    3.RoleGuard でメニュー制御        │
    │                    ロールに応じたメニューのみ表示      │
    │                    権限のないリンクは非表示           │
    │◀──メニュー画面──────────│                           │
    │                       │                           │
    │──業務画面へアクセス─────▶│                           │
    │                    4.JWT 検証＋ロール確認            │
    │                    Microsoft.Identity.Web が JWT 検証│
    │                    [Authorize(Roles=...)]           │
    │◀──業務画面──────────────│（確認あり）                 │
    │                       │                           │
    │◀──5.認証エラー画面──────│（確認無し・403 Forbidden）  │
    │   403 Forbidden        │                           │
```

**フロー説明：**

**(1) ログイン完了**  
Entra ID のログイン完了後、JWT に OID・UPN が含まれる。

**(2) UserRole テーブル照会**  
ログイン後に JWT の OID をキーに業務DB の UserRole テーブルを照会し、ロール一覧を ClaimsPrincipal に追加する。

**(3) RoleGuard でメニュー制御**  
RoleGuard でロールに応じたメニューのみ表示する。権限のないリンクはメニューに表示しない。

**(4) JWT 検証＋ロール確認**  
業務画面アクセス時、Microsoft.Identity.Web が JWT を自動検証し、ロール確認を行う（コード実装不要・自動処理）。

**(5) 権限無しの場合、403 Forbidden を返却し、認証エラー画面を表示する。**

## 5.10 Enterprise Application アクセス制御

本システムへの全部のユーザではなく、特定ユーザのみに制限するための Entra ID Enterprise Application の設定方針を記載する。

この制御は基盤側が担い、アプリ実装は不要である。

### 表5-6 App Registration と Enterprise Application の関係

| 項目 | App Registration | Enterprise Application |
|------|-----------------|----------------------|
| 役割 | アプリの身分証明書・開発設定 | ユーザへの見せ方・アクセス制御 |
| 作成タイミング | 基盤側が新規作成 | App Registration 作成時に自動生成 |
| 主な操作者 | 基盤側（開発側に連携）| 基盤側 |
| 設定内容 | クライアントID・リダイレクト URI・トークン設定 | ユーザ割り当て・グループ割り当て・SSO 設定 |
