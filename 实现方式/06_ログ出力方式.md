# 第6章 ログ出力方式

本章では、本システムで出力するログについて整理する。

---

## 6.1. 本システムで出力するログ

本システムで出力するログの種別と出力内容を以下に示す。

**表6-1　出力するログ一覧**

| 出力ログ | 出力内容 |
|---|---|
| ユーザ認証ログ | Entra ID へのログイン・トークン発行等の認証イベント |
| アプリ処理用ログ | 画面操作・ファイル入出力・API リクエスト/レスポンス・DB 接続が出力するログ |
| 外部連携用ログ | 格付判定システム・財務会計システム等の外部 API 通信、ファイル連携ログ |
| OCR 処理ログ | AnyForm OCR が出力した CSV ファイルの Blob Storage への転送・取込処理（CSV ファイル転送用バッチログ）が出力するログ |
| セキュリティ監査ログ | 不正アクセス試行・異常通信・VM セキュリティイベント等（JAFFIC 基盤が自動収集） |

※ アプリ処理用ログ・外部連携用ログは、全コードが同一アプリ内で動作するため、単一のログファイルに統合して出力する。  
※ ユーザ認証ログ・セキュリティ監査ログは JAFFIC 基盤が自動収集する。アプリ側の実装は不要。

---

## 6.2. ログ出力の目的

本システムでは、以下の目的からログの取得を行う。

**(1) セキュリティ**  
システムの不正アクセス、システム停止、情報漏えい、改竄の追跡を目的として監査用ログを記録し、システムのセキュリティを確保する。

**(2) 障害調査**  
システムにエラーが発生した場合、エラーの発生源を特定するためのトレース情報などを障害調査用ログとして記録し、障害解析の材料とする。

**(3) 稼働実績**  
システムのアクセス数、アクセス時間などの情報を稼働実績ログとして記録し、サービス利用状況の解析などを行う。

**(4) JAFFIC 基盤要件への対応**  
JAFFIC クラウドの運用規程に基づき、情報システム管理者が必要と認めた場合に対応するログを出力する。  
Azure Monitor・Log Analytics・Microsoft Sentinel との連携を前提とし、ログ収集・分析・セキュリティ監視の一元管理を実現する。

---

## 6.3. ログ出力方式

ログの出力は、出力先・出力箇所・出力内容に応じて出力方式が異なる。本システムのログ設計方針を以下に示す。

**ログ設計基本方針**

- アプリケーションログの実装は Serilog（.net 10）のみとし、DB へのログ書き込みは行わない。
- Serilog はログファイル（Azure VM）へ出力し、Azure Monitor エージェント（AMA）が Log Analytics Workspace へ自動転送する。
- ユーザ認証ログ（ログイン）は、Entra ID の診断設定により Log Analytics Workspace へ自動収集する。アプリ側の追加実装は不要。
- セキュリティ監視は JAFFIC 基盤が提供する Microsoft Sentinel が担う。

**表6-2　ログ出力方式**

| 出力ログ | 出力方式 | 出力先 | メンテナンス方法 |
|---|---|---|---|
| ユーザ認証ログ | Entra ID の診断設定により出力する。 | Log Analytics Workspace（Entra ID 設定により自動収集） | リテンションポリシー設定 |
| アプリ処理用ログ | Serilog はログファイル（Azure VM）へ出力する。 | ログファイル → AMA → Log Analytics Workspace | 設定による自動削除 |
| 外部連携用ログ | 同上 | ログファイル → AMA → Log Analytics Workspace | 同上 |
| OCR 処理ログ | バッチが出力する。 | OCR 端末で管理する。 | 同上 |
| セキュリティログ | JAFFIC 基盤が提供する Microsoft Sentinel が出力する。 | Log Analytics Workspace 経由 | リテンションポリシー設定 |

---

## 6.3. ユーザ認証ログ

ユーザ認証ログは、JAFFIC 共通テナントの Microsoft Entra ID が生成する認証イベントを Log Analytics Workspace へ自動収集することで取得する。アプリケーション側に追加実装は不要である。

収集される主な認証イベントを以下に示す。

- 正常ログイン
- 条件付きアクセスポリシーによるブロック
- トークン発行 / 説明
- アカウントロック　ユーザがアクセスした日時

---

## 6.4. アプリ処理用ログ・外部連携用ログ

アプリ処理・外部連携・OCR 処理ログの各ログは、Serilog（ASP.NET Core 統合）を用いてログファイルに出力し、Azure Monitor エージェント（AMA）が Log Analytics Workspace へ転送する。  
OCR 処理ログは処理端末に保存される。

### 6.4.1 出力ファイル構成

**表6-2　ログ出力箇所・ファイル種別に対するファイル名詳細**

| 出力箇所 | ファイル種別 | 保管場所（Azure VM） | ローテーション前ファイル名 |
|---|---|---|---|
| アプリ処理用ログ | アプリケーションログ | C:\Logs\app\ | app-yyyyMMdd.log |
| アプリ処理用ログ | デバッグログ | C:\Logs\app\ | debug_app-yyyyMMdd.log |
| OCR ファイル転送用バッチログ（OCR 端末） | アプリケーションログ | C:\Logs\batch\ | yyyyMMdd.log |

※ ファイル名の日付（yyyyMMdd）は Serilog の RollingFileSink が自動付与する。  
※ デバッグログは開発環境のみ有効。本番環境では出力しない。  
※ バッチログは Serilog を利用しない。

**表6-3　Log Analytics Workspace への転送先（AMA 設定）**

| ファイルパターン | Log Analytics カスタムテーブル | 備考 |
|---|---|---|
| C:\Logs\app\app-*.log | {System}_App_CL | オンライン処理 アプリケーションログである。 |

※ バッチログは OCR 端末で管理する。

### 6.4.2 ログ出力フォーマット

ログファイルの共通フォーマットを以下に示す。

**表6-4　共通フォーマット詳細**

| 出力項目 | 出力内容 |
|---|---|
| 出力日時 | ログ出力時刻（yyyy-MM-ddTHH:mm:ss.fffZ　ISO 8601 形式） |
| ログレベル | ログレベルを定義した値（表6-7 参照） |
| ホスト名 | 処理を実行したサーバのホスト名（Azure VM ホスト名） |
| クラス名 | ログを出力した .NET クラスの完全修飾名 |

**表6-5　アプリケーションログ フォーマット詳細**

| 出力項目 | 出力内容 |
|---|---|
| メッセージ ID | 業務でメッセージ単位に割り当てた ID（メッセージコード体系に準拠）である。 |
| メッセージ本文 | メッセージ ID に対するメッセージ（nrmList / wrnList / errList 区分に応じる）である。 |

### 6.4.3 ログレベルの定義

下記のログレベルを選定し、出力対象を振り分ける。

**表6-6　ログレベルと出力内容詳細**

| ログレベル | 出力条件 | 出力内容 | 開発時出力 | 運用時出力 |
|---|---|---|---|---|
| Fatal | 処理継続不可のエラー発生時 | 未捕捉例外（ExceptionHandlerMiddleware が捕捉） | ○ | ○ |
| Error | 処理継続可能なエラー発生時 | 業務エラー（errList に格納されるエラー） | ○ | ○ |
| Warning | 注意を促す動作発生時 | 業務警告（wrnList） | ○ | ○ |
| Information | 運用に必要な情報出力時 | 正常完了通知・外部 API 呼び出し結果（nrmList） | ○ | ○ |
| Debug | 開発時の詳細情報出力時 | SQL パラメータ詳細・EF Core クエリ内容 | ○ | − |

※ appsettings.json の Serilog:MinimumLevel で環境ごとに出力レベルを制御する（本番：Information、開発：Debug）。

**表6-7　各メッセージの出力例**

| ログレベル | メッセージ内容例 |
|---|---|

| Debug | 2026-01-01T09:00:00Z DEBUG API リクエスト受信。エンドポイント: {0}、ユーザー: {1} |

## 6.5. Serilog 実装方針

本システムのログ出力は Serilog（Serilog.AspNetCore）を用いて実装する。シンクはファイル出力のみとし、DB へのログ書き込みは行わない。

**表6-9　Serilog シンク（Sink）構成**

| シンク | 出力先 | 対象環境 | 備考 |
|---|---|---|---|
| File（RollingFile） | Azure VM のログファイル（C:\Logs\app\） | 全環境 | AMA が Log Analytics Workspace へ自動転送 |
| Console | 標準出力 | 開発環境のみ | Visual Studio / IIS Express デバッグ時の確認用 |

※ Serilog.Sinks.MSSqlServer は使用しない。DB へのログ保管は行わない。  
※ Serilog.AspNetCore の UseSerilogRequestLogging() により、全 HTTP リクエスト/レスポンスが自動的に Information ログとして記録される。  
※ ExceptionHandlerMiddleware（共通）が未捕捉例外を自動的に Fatal / Error ログとして出力する。

**表6-10　Serilog が提供するログ出力部品（Controller・Service クラス向け）**

| クラス / 使用方法 | メソッド | 開発時出力 | 運用時出力 | 説明 |
|---|---|---|---|---|
| ILogger\<T\>（DI 注入） | LogInformation() | ○ | ○ | 正常系・nrmList 対応 |
| ILogger\<T\>（DI 注入） | LogWarning() | ○ | ○ | 警告系・wrnList 対応 |
| ILogger\<T\>（DI 注入） | LogError() | ○ | ○ | エラー系・errList 対応 |
| ILogger\<T\>（DI 注入） | LogDebug() | ○ | − | デバッグ情報（開発環境のみ） |
| ExceptionHandlerMiddleware（共通基盤） | LogError() / LogCritical() | ○ | ○ | 未捕捉例外を自動出力 |
| Serilog RequestLoggingMiddleware（共通基盤） | 自動（全リクエスト） | ○ | ○ | HTTP リクエスト/レスポンス自動記録 |

## 6.6. ログローテーション方式

アプリ処理用ログ・外部連携用ログは、日付・ファイルサイズごとにローテーションを行い別ファイルとして管理する。Serilog の RollingFileSink により自動制御する。

① 日付ローテーション：ファイルの作成日付と現在日付が異なる場合  
→ ファイル名に作成日付が自動付与される（例：app-20251226.log）。新しい日付で新ファイルを生成して書き込みを行う。

② サイズローテーション：ファイルサイズが規定値を超えている場合  
→ 連番が付与された新ファイルを生成して書き込みを行う（例：app-20251226_001.log）。

**表6-11　ログのローテーション規定値（Serilog 設定）**

| 出力用途 | ファイル種別 | サイズローテーション |
|---|---|---|
| アプリ処理用ログ | アプリケーションログ | 20 MByte |
| アプリ処理用ログ | デバッグログ | 100 MByte（開発環境のみ） |
| 外部連携用ログ | アプリケーションログ | 20 MByte |
| 外部連携用ログ | デバッグログ | 100 MByte（開発環境のみ） |
| OCR 連携バッチログ | アプリケーションログ | 20 MByte |

※ Azure VM のログファイルは AMA が Log Analytics に転送後、7 日以内に削除する（ディスク節約）。  
※ Log Analytics Workspace のリテンション期間は本番環境で 90 日に設定する（デフォルト 30 日）。

**表6-12　ログ保管場所および保持期間**

| 保管期間 | 保管先 | 備考 | 対象ログ種別 |
|---|---|---|---|
| 0〜7 日間 | Azure VM ファイル + Log Analytics | ローカルは AMA 転送後に削除 | 全ログ |
| 7〜90 日間 | Log Analytics Workspace | 本番環境リテンション設定 | 全ログ |
| 90 日以上 | Log Analytics アーカイブ層 | 低コスト長期保管（検索コストあり） | 認証ログ・監査ログ |

## 6.7 ログ出力フロー概要

本システムにおけるログ出力の全体フローを以下に示す。

**【クライアント側（Angular 18 / ブラウザ）】**  
Entra ID ログイン → MSAL が Access Token 取得 → MsalInterceptor が全リクエストに自動付与  
→ 認証イベントは Entra ID 診断設定により Log Analytics（SigninLogs）へ自動収集

**【AP サーバ（Azure VM / IIS / .net 10）】**  
① Serilog RequestLoggingMiddleware → 全リクエスト/レスポンスを Information ログとしてファイル出力  
② ExceptionHandlerMiddleware → 未捕捉例外を Fatal / Error ログとしてファイル出力  
③ Controller → FluentValidation → Service（ILogger\<T\>.LogXxx()）→ ログファイル出力  
④ Polly リトライ発生時 → Warning ログをファイル出力

**【Azure 監視基盤（JAFFIC クラウド共通 / アプリ実装不要）】**  
⑤ Azure Monitor エージェント（AMA）→ VM ログファイルを Log Analytics Workspace へ自動転送  
⑥ Azure Monitor → VM / SQL MI メトリクス監視 → 閾値超過時アラート発砲  
⑦ Azure Network Watcher → NSG フローログ収集 → 異常通信検知  
⑧ Microsoft Sentinel → Log Analytics + Entra ID ログ統合 → SIEM 脅威分析  
⑨ Log Analytics → 90 日リテンション後、アーカイブ層へ自動移行
