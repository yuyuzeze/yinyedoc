# 第１２章　性能設計方式

本章は、新たなシステム（以下「本システム」）のリプレース開発にあたり、システム全体の性能に関する設計方針、性能目標値、各層における性能向上策、および性能テストの実施方法を定めるものである。

---

## 12.1 適用範囲

**表12-1　適用範囲**

| 対象区分 | 内容 |
|---------|------|
| 対象システム | 本システムの全サブシステム |
| 対象環境 | IIS + Azure SQL Database + Azure Blob Storage |
| 対象ユーザー | 信用基金（最大同時接続数：約40名） |

## 12.2 性能目標値

### 12.2.1 応答時間目標値

本システムの応答時間目標値を以下に示す。本目標値は要件定義「3.3 規模に関する事項」にて予想される最大負荷に対して50%の余裕を考慮して算定したものである。

目標値は、連携するシステムやデータベース等の状況に影響を受けることを踏まえ、必要に応じて目標値の見直し・信用基金様への提案・説明し、承認を得ること。

**表12-2　応答時間の目標値**

| 設定対象 | 指標名 | 目標値 | 補足 |
|---------|-------|-------|------|
| 画面遷移（初回表示・ページ切替） | レスポンスタイム | 3秒以内 | ピーク時を対象とする。 |
| 情報検索（一覧・絞込検索・集計） | レスポンスタイム | 5秒以内 | ピーク時を対象とする。 |
| 画面入力情報登録（新規・更新・削除） | レスポンスタイム | 3秒以内 | ピーク時を対象とする。 |
| 帳票イメージ表示（Excel / PDF） | レスポンスタイム | 10秒以内 | ピーク時を対象とする。 |
| ファイルアップロード（OCR連携CSVを含む） | 完了通知までの時間 | 10秒以内 | ファイルサイズ5MB以下を前提とする。 |
| 月次バッチ処理（保証料計算・統計資料等） | バッチ処理時間 | 30分以内 | 業務時間外実行（深夜帯）を想定。 |
| 外部API連携（格付判定システム等） | API応答時間（本システム側） | 5秒以内 | 外部システム処理時間を除く。タイムアウト設定：30秒。 |

### 12.2.4 応答時間測定条件

オンライン処理における応答時間の測定条件は(1)および(2)のとおりとする。

(1) 信用基金指定の2台以上の端末に、受注者の費用負担で納入する測定用ツールをインストールする等して、自動で端末での応答時間を測定し、その平均値を用いることとする。

(2) キャッシュ効果により応答時間が短縮されたと思われる測定結果は対象外とすること。

> ※ 応答時間の測定は「端末でのDOM Contentが完全表示されるまでの時間」を基準とし、ネットワーク遅延（VPN経由）を含めた実態に即した数値を用いること。

## 12.3 性能設計方針

本システムの性能目標値を達成するために、フロントエンド・バックエンド・データベース・インフラの各層において以下の性能設計方針を適用する。

---

### 12.3.1 フロントエンド性能設計

**表12-5　バンドルサイズ最適化**

| 設計項目 | 設計内容・方針 |
|---------|--------------|
| 遅延読み込み（Lazy Loading） | サブシステム単位でAngularモジュールを遅延読み込みする。ユーザーが該当サブシステムにアクセスした時のみバンドルチャンクを取得するため、初回表示速度を向上させる。 |
| AOT（Ahead-of-Time）コンパイル | プロダクションビルド時はAOTコンパイルを適用し、ブラウザ側のテンプレートコンパイル処理を不要とする。これにより初期画面描画時間を短縮する。 |
| Tree Shaking | 未使用のコード・モジュールをビルド時に自動除去する。import文の管理を徹底し、不要なAngular Materialモジュールをインポートしないよう実装規約として定める。 |
| 静的ファイル圧縮（gzip） | IIS側でgzip圧縮を有効化し、JavaScriptバンドル・CSSファイルを圧縮して転送量を削減する。JavaScript・CSSに対してgzip圧縮を適用し、転送サイズを60〜80%削減することを目標とする。 |
| ブラウザキャッシュ制御 | Angular CLIが生成するコンテントハッシュ付きファイル名（例：main.a1b2c3.js）を活用し、変更のないファイルはブラウザキャッシュから読み込む。ただし性能測定時はキャッシュを無効化（キャッシュクリア状態）にて計測する。 |

**表12-6　画面描画・インタラクション最適化**

| 設計項目 | 設計内容・方針 |
|---------|--------------|
| OnPush変更検知戦略 | 全コンポーネントでChangeDetectionStrategy.OnPushを採用する。Angular SignalsおよびRxJSのObservableとの組み合わせにより、不要な変更検知サイクルを排除し、DOM更新コストを最小化する。 |
| 仮想スクロール（Virtual Scroll） | 大量データの一覧表示において、Angular CDKのVirtualScrollViewportを使用する。DOM上に描画するのは表示領域内の行のみとし、数百〜数千件のデータ表示時もスクロール性能を維持する。 |
| ローディングインジケーター | LoadingInterceptorによりAPIリクエスト中はプログレスバーを表示する。ユーザーの待機感を軽減し、処理中であることを視覚的に伝える。画面が応答しているように見せることで体感応答時間を改善する。 |
| ページネーション（サーバーサイド） | 全件取得を禁止し、一覧取得APIはページ番号・ページサイズをパラメータとして受け取るサーバーサイドページネーションを必須とする。 |
| 入力フォームの非同期バリデーション制御 | サーバーサイドバリデーションが必要な項目（重複チェック等）は、debounceTime（300ms）を設定して連続入力中のリクエスト発行を抑制する。 |

---

### 12.3.2 バックエンド性能設計

**表12-7　非同期処理**

| 設計項目 | 設計内容・方針 |
|---------|--------------|
| async/await の全面採用 | Controller・Service・Repositoryの全メソッドでasync/awaitを使用し、I/O待機中のスレッドブロッキングを排除する。EF CoreのQueryAsync・SaveChangesAsyncを必ず使用し、同期APIの使用を禁止する。 |
| BackgroundServiceによるバッチ処理分離 | 長時間かかる処理はIHostedService（BackgroundService）として非同期実行する。HTTPリクエストのタイムアウトとは独立して実行し、フロントエンドには処理受付結果を即時返却する。 |
| CancellationToken の伝播 | 全APIエンドポイントからCancellationTokenを受け取り、EF Coreクエリ・外部APIコール・ファイルIOまで伝播させる。クライアントが接続を切断した場合に不要な処理を速やかに中断する。 |

**表12-8　レスポンスキャッシュ・メモリ管理**

| 設計項目 | 設計内容・方針 |
|---------|--------------|
| IMemoryCacheによるインメモリキャッシュ | 更新頻度の低いマスターデータ（コードマスタ・部門マスタ・保証種別マスタ等）はIMemoryCacheに保持し、DBアクセスを削減する。キャッシュ有効期間は原則30分とし、マスターデータ更新時にキャッシュを無効化する仕組みを設ける。 |
| レスポンス圧縮 | ASP.NET Core組み込みのResponseCompression Middlewareを有効化し、大量データを返すAPI（一覧取得・帳票データ等）の転送量を削減する。Content-Type: application/jsonに対してBrotli圧縮（第1優先）またはgzip圧縮を適用する。 |
| 帳票生成のストリーム出力 | ExcelCreator v12による帳票生成はMemoryStreamへ直接書き出し、FileContentResultとしてストリーミング返却する。帳票データをいったんファイルシステムやBlobに保存する不要なI/Oを排除する。 |
| Pollyによる外部API耐障害設計 | 格付判定システム等の外部APIへのコールはPollyのRetryPolicy（指数バックオフ、最大N回）とCircuitBreakerPolicy（N回失敗で30秒遮断）を組み合わせる。外部APIの応答遅延が本システムのスレッドプールを枯渇させないようタイムアウトを設定する。 |

**表12-9　ページネーション・データ量制限**

| 設計項目 | 設計内容・方針 |
|---------|--------------|
| 全件取得の禁止 | 一覧取得APIは必ずページネーションパラメータ（pageNumber, pageSize）を受け取る設計とする。pageSize上限を設定し、上限を超えるリクエストは400 Bad Requestを返す。 |
| SELECT N+1問題の排除 | EF Coreのクエリ設計時にNavigationPropertyのN+1問題を防止する。Include / ThenInclude による積極的読み込みまたは明示的読み込みを適切に使い分け、発行SQLを最小化する。本番リリース前にSerilogでSQL文をDEBUGレベルで出力し、N+1が発生していないことをレビューで確認する。 |
| 射影（Select / DTO変換） | クエリ時にEntityを全列取得（SELECT *）せず、必要な列のみをDTOに射影する。AutoMapperのProjectTo\<T\>()を使用してDB側でSELECT列を絞り込む。 |

## 12.4 データベース性能設計

**表12-10　インデックス設計**

| 設計項目 | 設計内容・方針 |
|---------|--------------|
| 主キー・外部キーインデックス | 全テーブルに主キー（CLUSTERED）を設定する。外部キー列には非クラスター化インデックス（NONCLUSTERED）を付与し、JOIN性能を確保する。 |
| 検索条件カラムへのインデックス付与 | 頻繁に検索条件となる列（法人番号・保証番号・処理年月・ステータス等）には非クラスター化インデックスを設計段階で設ける。複合検索条件が多い画面は複合インデックスの有効性を検証する。 |
| インデックス過多の抑制 | インデックスは書き込み性能に影響するため、DML（INSERT/UPDATE/DELETE）の多いテーブルへの過剰なインデックス付与を避ける。実際のクエリパターンを分析した上で必要最小限のインデックスを設計する。 |
| 実行計画レビュー | 性能テスト実施時にAzure SQL DatabaseのQuery Performance Insightを使用し、高コストクエリを抽出・分析する。 |

**表12-11　接続・クエリ最適化**

| 設計項目 | 設計内容・方針 |
|---------|--------------|
| 接続プーリング | EF Core + Azure SQL DatabaseはADO.NETの接続プーリングを自動使用する。接続文字列のMin Pool Size / Max Pool Sizeを適切に設定し、接続生成コストを削減する。 |
| AsNoTracking の適用 | 参照系クエリ（SELECT）にはAsNoTracking()を付与し、EF Coreの変更追跡オーバーヘッドを排除する。変更追跡が不要な読み取り専用クエリ（一覧表示・帳票データ取得等）は全てAsNoTracking()を適用することをコーディング規約として定める。 |
| 一括更新（Bulk Operation） | 月次バッチ処理等で大量レコードを更新する場合、EF Coreの1件ずつSaveChangesは使用しない。ExecuteUpdateAsync / ExecuteDeleteAsyncまたはBulk Insert用ライブラリを採用し、DB往復回数を最小化する。 |
| コンパイル済みクエリ | 同一クエリを繰り返し実行する処理にはEF Coreのコンパイル済みクエリを適用し、クエリ翻訳コストを削減する。 |
