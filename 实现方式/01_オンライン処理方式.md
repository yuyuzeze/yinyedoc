## 1.1 処理概要

新たなシステム（以下「本システム」）は、Microsoft Azure クラウドプラットフォームが提供する各種マネージドサービス（Azure VM、Azure Blob Storage、Azure SQL Database 等）を基盤として実現する。

SPA モデルと REST API を組み合わせた構成を採用し、インターフェースと業務ロジックの処理部分を明確に分離したアーキテクチャを採用する。

画面表示部分には Angular を採用し、業務ロジックの処理部分には .NET Core Web API を採用する。両者は同一の IIS サイト上に配置され、一つの Web アプリケーションとして動作する。

※ Single Page Application（SPA）モデルとは、Web アプリケーションにおいて、初回表示時に必要なページを読み込んだ後、画面遷移のたびにページ全体を再読み込みするのではなく、必要なデータや画面要素のみを動的に更新して動作する方式を指す。

### フロントエンド（SPA）側では：

ユーザーインターフェースの表示、ユーザーとのインタラクションの処理、バックエンド（REST API）から返却したデータの一時保存を行う。

Angular と TypeScript を基盤として開発する。画面のインターフェースには Angular Material を使用してシステムの設計レイアウトに合わせてカスタマイズする。

本システムは Angular による完全な SPA 構成であり、フォーム送信およびデータ連携は HttpClient を用いた非同期通信にて行う。

### バックエンド（REST API）側では：

SPA からのリクエストを受け取り、業務ロジックの処理を行い、JSON 形式のデータを返却し、SPA 側でリアルタイムに表示処理を行う。

.NET フレームワークを基盤とし、具体的には .NET Core Web API を利用して、REST API を迅速かつ構造的に実装する。

データアクセス層には Entity Framework Core を ORM として採用し、効率的な CRUD 操作を実現する。

Microsoft Entra ID と連携し、JWT（JSON Web Token）を用いた認証・アクセス権限管理（RBAC）を行い、データフローの安全性を確保する。

また、セキュリティ確保のため、通信は HTTPS プロトコルを使用する。

### 本アーキテクチャ採用の導入効果：

- 初回ロード以降はバックエンドと純粋なデータ（軽量な JSON 形式）のみをやり取りする。基本的に HTML のフルリロードをしないため、ネットワーク通信量が削減され、ページのレスポンスタイムが向上する。
- 画面遷移は Angular Router によって制御されるため、ブラウザ戻る時やリロード時における POST の再送信や意図しないエラーは従来のフォーム POST 送信（画面遷移）に比べて起きにくくなる。
- 画面技術を旧システム（ASP.NET Web Forms）から SPA（Angular）へ刷新することで、画面操作の応答性が向上し、ユーザー体験が改善される。
- フロントエンドとバックエンドの定義済みの OpenAPI（Swagger）インターフェースのみで JSON 形式で連携するため、フロントエンド・バックエンドを並行開発でき、スケジュールの短縮になる。

### 本アーキテクチャのデメリットと解決策：

初回ロードは遅くなるが、Lazy Loading で通信を分割、初回以降はキャッシュを活用し、初回ロード遅延を軽減する。
