# 第１３章 拡張性設計方式

本章は、新たなシステム（以下「本システム」）の性能拡張性および機能拡張性に関する設計方針を定めるものである。

---

## 13.1 性能の拡張性設計

本システムはJAFFICクラウド（Microsoft Azure）上のマネージドサービスを最大限活用することで、利用率やデータ量の増加に対し、アプリケーションコードを変更することなく性能を拡張可能な設計とする。  
拡張性要件は定義せず、Azureの動的スケーリング機構（VMサイズ変更・SQL層変更）により対応する。

### 表13-1 Azureリソースのスケーリング方針

| Azureリソース種別 | スケーリング手段 | 操作・手順 |
|---|---|---|
| Azure VM（APサーバ） | 垂直スケールアップ（VMサイズ変更） | Azure Portal / Azure CLI にて VMサイズを変更。IIS / Kestrel 設定変更は不要となる。 |
| Azure SQL Database | サービス層変更（DTU / vCore 変更） | Azure Portal の「コンピューティングとストレージ」にて SKU をアップグレード。接続文字列の変更不要となる。 |
| Azure Blob Storage | 容量自動拡張（上限なし） | Azure Blob Storage は自動拡張のため事前設定不要となる。 |
| Log Analytics Workspace | データ保持期間延長・取込量上限変更 | Azure Portal の「使用量と推定コスト」で設定する。 |

## 13.2 性能の拡張性設計

本システムは以下のアーキテクチャ設計により、新機能追加・外部連携拡張が低コスト・低リスクで実施できる構造とする。

### 13.2.1 フロントエンド

- サブシステム単位の Lazy Loading によりモジュール独立性を確保。新規サブシステムの追加は既存モジュールへの影響なく実施可能である。
- サブシステム + AUTH の構造を基本とし、将来サブシステムは同一パターンで追加可能である。
- RoleGuard / MsalInterceptor 等の共通処理は DI により自動適用。新機能追加時に個別実装不要となる。
- Angular Material コンポーネントの統一使用により、UI 追加時のデザイン一貫性を確保する。

### 13.2.2 バックエンド

- Clean Architecture（Api / Application / Domain / Infrastructure の4層分離）により業務ロジックとインフラの結合を最小化する。
- Repository\<T\> + UnitOfWork パターンにより、新テーブル追加時はドメインエンティティと Repository を定義するだけで CRUD が実装可能である。
- 外部 API 連携は HttpClientFactory + Polly により実装。新規外部システム連携の追加は新 Service クラスの追加のみで対応可能である。

### 13.2.3 コンポーネントの再利用性確保

利用者ニーズおよび業務環境の変化に最小コストで対応するため、以下の共通コンポーネント体系を整備する。

### 表13-2 共通コンポーネント体系一覧

| 区分 | 共通コンポーネント名 | 用途・役割 | 再利用方法 |
|---|---|---|---|
| Angular | MsalInterceptor | Entra ID Access Token 自動付与 | 全サブシステムで DI 登録済み。個別実装不要 |
| Angular | ErrorInterceptor | 401/403/500 統一エラー処理・Toast通知 | 全サブシステムで DI 登録済み。個別実装不要 |
| Angular | RoleGuard | メニュー・画面単位の権限制御 | route に canActivate で付与。ロールコード指定のみ |
| Angular | NotificationService | nrmList/wrnList/errList の Toast 表示 | 画面 TS から inject して呼び出し |
| Angular | BaseListComponent | 一覧画面基底クラス（検索・ページング・エクスポート） | 各一覧画面コンポーネントが extends |
| Angular | BaseFormComponent | 入力画面基底クラス（バリデーション・保存） | 各入力画面コンポーネントが extends |
| .NET | BaseApiController | Controller 基底クラス（ApiResponse 統一変換） | 全 Controller が継承 |
| .NET | ExceptionHandlerMiddleware | 全例外外キャッチ → errList[ESYS50001] | Program.cs で一度登録。個別実装不要 |
| .NET | Repository\<T\> / UnitOfWork | EF Core の汎用 CRUD ラッパー | 各ドメインの Repository が継承 |
| .NET | ServiceResult\<T\> | Service 層の統一戻り値パターン | Ok() / OkWithWarning() ファクトリメソッド使用 |
| .NET | FluentValidation AbstractValidator\<T\> | リクエスト DTO の入力検証 | DTO ごとに派生クラスを作成・DI 登録 |

### 13.2.4 コンポーネントの再利用性確保

機能・画面・帳票・API における固有の ID・項目名等は、以下の系統だった命名ポリシーに従い、中長期的な重複を避けるとともに可読性を確保する。

### 表13-3 命名規則ポリシー一覧

| 対象 | 命名規則 | 形式例 | 補足・禁止事項 |
|---|---|---|---|
| API エンドポイント | `/api/<subsystem>/<screen>/<method>` | `/api/hosho/hosho1010/search` | サブシステム名は小文字英字。番号は4桁固定 |
| APIメソッド名（URL末尾） | lowerCamelCase 英語（日本語ローマ字可） | `jikoToroku`, `shodanSearch` | スネークケース・大文字始まり禁止 |
| API 番号（内部ID） | `<サブシステム大文字><4桁番号><メソッド名>` | `HOSHO1010Search` | 重複禁止。廃番は欠番のまま残す |
| エラー/通知コード | `MSG{SYS}Ixxxx` / `MSG{SYS}Exxxx` | `MSGAE0001` | 番号帯は設計書で一元管理 |
| Angular コンポーネント | PascalCase + 役割サフィックス | `Hosho1010SearchComponent` | 機能追加時に既存名称と重複しないこと |
| Angular サービス | PascalCase + Service サフィックス | `Hosho1010Service` | サブシステム別ディレクトリに配置 |
| .NET Controller クラス | PascalCase + Controller サフィックス | `Hosho1010Controller` | サブシステムフォルダ（`Controllers/Hosho/`）配置 |
| DB テーブル名 | `T_<システム名>_<業務名>` | `T_HOSHO_BIKO` | プレフィックス等の体系は DB 設計書で一元管理 |
| Azure リソース名 | `<種別略称>-<システム>-<環境>` | `vm-forestry-prod` | JAFFIC 命名規則に準拠すること |
