本章では、本システムにおいてREST APIのリクエストおよびレスポンスを生成方法について説明する。

## 4.1. REST APIのリクエストボディ

### 表4-1 リクエスト定義の詳細情報

| 定義 | 内容 |
|------|------|
| **HTTPメソッド** | POSTメソッドを使用する。 |
| **コンテンツタイプ** | ファイルをアップロードする場合、content-typeをmultipart/form-dataに設定する。 |
| | 上記以外の場合、content-typeをapplication/jsonに設定する。 |
| **リクエストボディ** | バックエンドアプリケーション（REST API）の要件に基づいて、必要な属性を選択し、各APIに対応するリクエストボディを作成する。 |
| | リクエストボディには、REST APIに必要な属性を含める。 |
| | REST APIで入力パラメータが不要な場合、リクエストボディを空のオブジェクトとして設定する。 |
| **認証ヘッダー** | @azure/msal-angularのMsalInterceptorがAccess Tokenを自動付与する。 |
| | 手動でのAuthorizationのヘッダー設定は不要（Interceptorが自動処理）となる。 |

### リクエストボディの例

**（例1）scrxxxxx/PageInit（平時-保証審査一覧初期化）**

- HTTPメソッド：POST
- コンテンツタイプ：application/json
- リクエストボディ：なし
  - 属性なしの空オブジェクトを作成する。

```json
{}
```

**（例2）scrboxxx/GuarantorSearch（有事-保証人情報取得）**

- HTTPメソッド：POST
- コンテンツタイプ：application/json
- リクエストボディ：あり
  - リクエストボディ生成時にデータ検索を行うために、kanjiNameとtelNoの2つのパラメータを設定する。

```json
{
  "kanjiName": "〇〇",
  "telNo": "07012345678"
}
```

## 4.2. 共通のレスポンスボディ

REST API形式へ変換するため、レスポンスボディを以下のように共通的に定義する。

### 表4-1 レスポンス定義の詳細情報

| パラメータ名 | 内容 |
|------------|------|
| `result` | API呼び出し状態が成功の場合、返却したレスポンスを格納する。 |
| `messages` | レスポンスメッセージを格納する。 |
| `　nrmList` | 正常状態のメッセージ・コードを格納する。 |
| `　　code` | 該当するメッセージコード |
| `　　message` | 該当するメッセージ |
| `　wrnList` | 警告状態のメッセージ・コードを格納する。 |
| `　　code` | 該当するメッセージコード |
| `　　message` | 該当するメッセージ |
| `　errList` | エラー状態のメッセージ・コードを格納する。 |
| `　　code` | 該当するメッセージコード |
| `　　message` | 該当するメッセージ |
| `statusdetailmessage` | 該当するステータス詳細メッセージを格納する。 |
| `statusCode` | 該当するHTTPステータス（200、204、400など）の情報を格納する。 |

## 4.3. HTTPステータスの定義

システムで使用するHTTPステータスコードを定義する。  
API側のHTTPステータスコードのレスポンスおよび画面側がAPIからの応答を受け取った際の処理を標準化する。

### 表4-3 HTTPステータスコード一覧

| HTTPステータス | 説明 | 使用ケース | バックエンドでの処理状況 | フロントエンドでの処理状況 |
|-------------|------|----------|----------------------|------------------------|
| **200** | 正常終了 | APIの呼び出し状態が成功であり、返却データありの場合。 | データ取得が成功する。 | UIにデータを表示し、状態を更新する。 |
| **201** | 正常終了（新規作成に成功） | APIが正常終了した（新規作成が成功した）場合。 | 新規作成が成功する。 | 成功メッセージを表示し、リダイレクトまたは一覧を更新する。 |
| **202** | 正常終了（非同期受理） | APIが非同期APIにリクエスト送信に成功した場合。 | HTTPステータスでリクエストが受理されたことを返す。 | 成功メッセージを表示し、リダイレクトまたは一覧を更新する。 |
| **204** | 正常終了（対象データが存在しない/削除に成功） | APIの呼び出し状態が成功であるが、データが存在しない場合。 | 削除が成功する、またはデータ返却不要で更新する。 | UIから削除し、データの更新は不要である。 |
| **400** | 異常終了（バリデーションデータ不正） | 当処理/バリデーション等の処理を終了した場合。 | 入力データが不正である（例：バリデーション、JSON形式）。 | フォームエラーや具体的なメッセージを表示する。 |
| **401** | 異常終了（未認証） | 認証エラーが発生した（ログイン失敗またはAPIキー無効）場合。 | トークンが期限切れ、未ログイン、またはAPIキーが不正である。 | ログイン画面へ遷移し、メッセージを表示する。 |
| **403** | 異常終了（権限なし） | ログインに成功したが権限不足によりコンテンツへアクセスできない場合。 | ログイン済みだがアクセス権がない。 | 「権限なし」メッセージを表示し、機能を非表示にする。 |
| **404** | 異常終了（URLで指定した対象が存在しない） | 有効なURLが見つからなかった場合。 | リソースやエンドポイントが見つからない。 | 404エラー画面または「見つからない」メッセージを表示する。 |
| **405** | 異常終了（許可されないHTTPメソッド） | HTTPメソッドの不正により異常終了した（Method not allowed）場合。 | HTTPメソッドを誤って呼び出す（例：GETではなくPOSTを送信する）。 | 技術的なエラーメッセージを表示し、必要に応じてエラーレポートを送信する。 |
| **408** | リクエストタイムアウト | サーバーがクライアントからの接続を受け付けたが、クライアントがサーバーの許容時間内にHTTPリクエストの全データを送信しなかった場合。 | HTTPステータスコード408を返す。 | 対応する重大なエラー画面を表示する。 |
| **409** | 異常終了（排他エラー） | 他のリクエストと重複してデータの挿入や更新をリクエストした（Exclusive error）場合。 | データが重複している、または更新に適さない状態である。 | 「データは既に存在する」や「競合」メッセージを表示する。 |
| **415** | 異常終了（受付不能なデータ形式） | 送信されたデータ種別が未対応である（Unsupported Media Type）場合。 | ContentTypeを誤って送信する（例：JSONではなくXMLを送信する）。 | 技術的なエラーを表示し、データ送信設定を再確認する。 |
| **500** | 異常終了（サーバ内部エラー） | 業務エラー、例外、システムエラー等が発生した場合。 | バックエンド処理で予期しないエラーが発生する。 | 対応する重大なエラー画面を表示する。 |
| **502** | 異常終了（不正なゲートウェイ） | ゲートウェイまたはプロキシが不正な応答を受信した場合。 | ゲートウェイがバックエンドからエラー応答を受信する。 | 「サーバーへの接続エラー」メッセージを表示する。 |
| **503** | 異常終了（サービス利用不可） | 業務エラー、例外、システムエラー等が発生した場合。 | サーバーがメンテナンス中または過負荷状態である。 | 「システムメンテナンス中」メッセージを表示する。 |
| **504** | 異常終了（タイムアウト） | ゲートウェイまたはプロキシが応答のタイムアウトになった場合。 | ゲートウェイが期限内に応答を受信できない。 | 「タイムアウト」メッセージを表示し、再試行を許可する。 |

### 4.3.1. APIの呼び出し結果でHTTPステータスを200で返却した場合

API呼び出し状態が成功であり、HTTPステータスを200で返却した場合、返却したデータをパラメータのresultに格納する。

**（例）API「scrxxxxx/guarantorLoad」保証人情報取込の呼び出しに成功した場合**

Response body

```json
{
  "result": {
    "list": [
      {
        "hosyoNO": "06-7-0887",
        "syusshiCD": "23-2-126",
        "skname": "有限会社△△",
        "K_hkind": "管理中止"
      },
      {
        "hosyoNO": "11-ネ-0284",
        "syusshiCD": "01-2-264",
        "skname": "有限会社〇〇",
        "K_hkind": "通常"
      }
    ]
  },
  "messages": null,
  "statusdetailmessage": "OK",
  "statusCode": "200"
}
```

---

### 4.3.2. APIの呼び出し結果でHTTPステータスを204で返却した場合

API呼び出し状態が成功であるが、データが存在しない場合、HTTPステータスを204で返却する。

**（例）API「scrxxxxx/guarantorLoad」保証人情報取込の呼び出し結果でデータが返却されなかった場合**

Response body

```json
{
  "result": {
    "list": []
  },
  "messages": null,
  "statusdetailmessage": "No data",
  "statusCode": "204"
}
```

---

### 4.3.3. APIの呼び出し結果でHTTPステータスを500で返却した場合

API呼び出しでバリデーションチェックエラーが発生した場合、HTTPステータスを500で返却する。

**（例）API「scrxxxxx/rakingExecute」随時格付の呼び出しでバリデーションチェックエラーが発生した場合**

Response body

```json
{
  "result": null,
  "messages": {
    "nrmList": [],
    "wrnList": [],
    "errList": [
      {
        "code": "E〇〇〇〇〇",
        "message": "自動審査に失敗しました。"
      }
    ]
  },
  "statusdetailmessage": "InternalError",
  "statusCode": "500"
}
```

## 4.4. URI設計方針

次期システムについては、以下の標準に従いURIを設計する。

```
https://<website>/<subsystem>/<screen name>/<method name>
```

### 詳細

#### `<website>`

- 本番環境のベースURLである。
- 画面のサブシステム略称を小文字で示す。（例：ho（保証）/sa（査定））
- すべての文字を小文字で示すこと。
- 特殊文字を使用しないこと。

| フェーズ | website | 備考 |
|---------|---------|------|
| 第1段階（VM + IIS） | `https://system.gov.local` | 政府内部ネットワーク（VPN経由のみアクセス可） |
| 第2段階（PaaS） | `https://api.gov-internal.go.jp` | ※要顧客確認・仮定義 |
| ローカル開発 | `http://localhost:4200` | proxy.conf.jsonで `/api` → `localhost:5001` に転送 |

#### `<subsystem>`

- 画面のサブシステム略称を小文字で示す。（例：ho（保証）/sa（査定））
- すべての文字を小文字で示すこと。
- 特殊文字を使用しないこと。

| サブシステム | URI | Angularタブ | 旧フォルダ対応 | 由来 |
|------------|-----|------------|--------------|------|
| HOSHO（保証） | `/hosho` | 保証タブ | `hoshou/shinsa/` | 旧システム引継ぎ |
| SATEI（査定） | `/satei` | 査定タブ | `hoshou/kanri/` | 旧システム引継ぎ |
| SUISHIN（推進） | `/suishin` | 推進タブ | 推進フォルダ | 旧システム引継ぎ |
| KEIRI（経理） | `/keiri` | 経理タブ | `keiri/` | 旧システム引継ぎ |
| KYOTSU（共通） | `/kyotsu` | 共通タブ | `kyoutsu/` | 旧システム引継ぎ |
| KASHI（貸付）★新規 | `/kashi` | 推進タブ内 | 該当なし | 今回新規追加 |
| KITAKU（寄託）★新規 | `/kitaku` | 推進タブ内 | 該当なし | 今回新規追加 |
| AUTH（認証） | `/auth` | 全業務共通 | — | 新規 |

#### `<screen name>`

- 画面IDを小文字の形式で示す。（例：scrb0040　事故登録画面）
- すべての文字を小文字で示すこと。
- 特殊文字（記号・スペース等）を使用しないこと。

#### `<method name>`

- APIの操作種別を英名詞+英単語で示す。（例：PageInit / ClaimSearch / ClaimCreate / ClaimDelete / ReportExport / FileImport 等）
- すべての文字を小文字で示すこと。
- 特殊文字（記号・スペース等）を使用しないこと。

### メソッド名 命名規則（lowerCamelCase）

1画面に複数の異なる操作ボタンが存在する場合、`create` / `update` 等の汎用動詞のみでは区別できないため、`<名詞><動詞>` 形式の lowerCamelCase を採用する。

| 操作 | メソッド名（例） |
|------|--------------|
| 画面初期化 | `pageInit` |
| 一覧検索 | `xxxSearch`（例: claimSearch） |
| 新規登録 | `xxxCreate`（例: claimCreate） |
| 更新 | `xxxUpdate`（例: claimUpdate） |
| 削除 | `xxxDelete`（例: claimDelete） |
| 帳票出力 | `xxxExport`（例: reportExport） |
| ファイル取込 | `xxxImport`（例: fileImport） |
| 保証人情報取得 | `guarantorLoad` |
| 格付実行 | `rakingExecute` |

### URIの例

```
https://api.gov-internal.go.jp/hosho/scrb0040/pageInit
https://api.gov-internal.go.jp/hosho/scrb0040/claimSearch
https://api.gov-internal.go.jp/satei/scrb0040/guarantorLoad
https://api.gov-internal.go.jp/satei/scrb0040/rakingExecute
https://api.gov-internal.go.jp/kashi/scrk0010/claimCreate
https://api.gov-internal.go.jp/kitaku/scrt0010/claimCreate
```

ローカル開発時（proxy経由）：

```
/api/hosho/scrb0040/pageInit
/api/hosho/scrb0040/claimSearch
/api/satei/scrb0040/guarantorLoad
```
