# 第8章 ファイルアップロード・ダウンロード方式

本章では、本システムにおけるファイルアップロード・ダウンロード方式を定義する。

本システムは Angular フロントエンドと .NET Web API バックエンドを Azure VM + IIS 上に同一オリジンで配置する構成を採用する。  
すべてのファイルは Azure Blob Storage に保存し、AP サーバーのローカルディスクにファイルを保存する構成は採用しない。  
Blob Storage へのアクセスは .NET API のみが行い、クライアント（Angular）に対して SAS Token を直接公開しない。AP サーバーは Managed Identity を使用して Blob Storage に認証する。

---

## 8.1. ファイル操作の全体像

本システムにおけるファイル操作は以下の3種類のアップロード経路と1種類のダウンロードで構成する。

- **【アップロード①】業務画面アップロード**：画面から Excel ファイルまたは各種添付ファイルをアップロードし、永続化時に Blob Storage の `app` コンテナに保存する。
- **【アップロード②】財務連携ファイル取込**：財務会計システムが `finance` コンテナに配置したファイルを、入金実績画面で読み込む。
- **【アップロード③】OCR ファイル取込**：AnyForm OCR がオンプレミスで出力した CSV を `ocr` コンテナに配置し、OCR ファイル確認画面で読み込む。
- **【ダウンロード】帳票出力**：業務処理の結果として PDF / Excel / CSV の帳票を生成してダウンロードする。

---

### 8.1.1. Azure Blob Storage コンテナ構成

本システムで使用する Azure Blob Storage のコンテナ構成は以下のとおり。各コンテナはアクセス権限・用途を明確に分離する。

**表8-1 Azure Blob Storage コンテナ構成**

| コンテナ名 | 経路 | アクセス方式 | 用途・備考 |
|---|---|---|---|
| `app` | ① | API 経由（Managed Identity） | 業務画面からアップロードされた添付ファイルを保管。パス構造：`/uploads/{subsystem}/{screen}/{guid}.{ext}` |
| `finance` | ② | 財務システム→Blob API 読取のみ | 財務会計システムが SFTP 経由で IF ファイルを配置する受信専用コンテナ。パス構造：`/keirisuser/NYUKIN/` |
| `ocr` | ③ | OCR 端末→Blob API 読取 | AnyForm OCR 出力 CSV の受信専用コンテナ。OCR 専用画面に反映する。 |

---

## 8.2. アップロード方式

### 8.2.1. ファイルアップロード仕様

ファイルのアップロードは .NET Core の IFormFile / IFormFileCollection を使用して実現する。クライアント（Angular）から Content-Type: multipart/form-data でリクエストを送信し、アップロードされたファイル情報が IFormFile オブジェクトとして Controller のアクションメソッドに渡される。

アップロードされたファイルは `IFormFile.OpenReadStream()` で取得したストリームを `Azure.Storage.Blobs` SDK の `BlobClient.UploadAsync()` を使用して Azure Blob Storage へ直接書き込む。AP サーバーのローカルディスクには一時ファイルを作成しない。

---

### 8.2.2. ファイルアップロードの定義値

ファイルアップロードに関する設定値は appsettings.json の FileUpload セクションで一元管理する。

**表8-2 ファイルアップロード定義値**

| 定義名称 | appsettings.json キー | 設定値 | 説明 |
|---|---|---|---|
| 最大ファイル数 | `FileUpload:MaxFileCount` | 20 | IFormFileCollection に含まれるファイル数の上限。 |
| Blob コンテナ名 | `FileUpload:BlobContainerName` | app | アップロード先コンテナ名。サブシステム別プレフィックスで管理する。 |
| アップロード並行数 | `FileUpload:MaxConcurrency` | 8 | BlobUploadOptions.TransferOptions.MaximumConcurrency に設定する並列転送スレッド数。 |
| チャンクサイズ | `FileUpload:BlockSizeBytes` | 4,194,304 | BlobUploadOptions.TransferOptions.InitialTransferSize に設定する分割転送ブロックサイズ（4 MB）。 |

---

### 8.2.3. ファイルタイプ別バリデーター

ファイルタイプごとに FluentValidation の `AbstractValidator<IFormFile>` を継承したバリデータークラスを定義し、拡張子・サイズのチェックを一元管理する。  
各バリデーターのサイズ上限値は appsettings.json で変更可能となる。

**表8-3 ファイルタイプ別バリデーター定義**

| バリデータークラス | 対象ファイルタイプ | サイズ上限 | 許可拡張子 |
|---|---|---|---|
| `BinaryFileValidator` | バイナリ（画像） | 50 MB | `.gif` / `.jpg` / `.jpeg` / `.jpe` / `.jfif` / `.tif` / `.png` |
| `CsvFileValidator` | CSV | 50 MB | `.csv` |
| `PdfFileValidator` | PDF | 100 MB | `.pdf` |
| `ExcelFileValidator` | Excel | 100 MB | `.xlsx` / `.xls` |
| `DocFileValidator` | ドキュメント | 50 MB | `.doc` |

---

### 8.2.4. ファイルアップロード時のチェック

ファイルを含むリクエストに対して以下のチェックを実施する。

**表8-4 ファイルアップロード時チェック処理詳細**

| チェック処理 | 説明 |
|---|---|
| 拡張子チェック | 許可されていない拡張子の場合、errList にエラーメッセージを格納し HTTP 400 を返す。 |
| Content-Type チェック | リクエストの Content-Type とファイル拡張子の整合性を検証する。<br>・`.xls` / `.xlsx` → `application/vnd.ms-excel` または `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`<br>・`.pdf` → `application/pdf`<br>・`.csv` → `text/csv` または `text/plain`<br>・画像 → `image/*`<br>不整合の場合は errList にエラーメッセージを格納し HTTP 400 を返す。 |
| ファイルサイズチェック | `RequestSizeLimitAttribute` でリクエスト全体サイズを制限し、`IFormFile.Length` を表8-3 の上限値と照合する。超過した場合はエラーメッセージを返す。 |
| 文字コードチェック | ファイル名に UNICODE→SHIFT-JIS（CP932）変換不可能な文字が含まれていないかチェックする。変換不可の場合はエラーメッセージを返す。 |
| パストラバーサル対策 | `Path.GetFileName()` でトラバーサル文字（`../` 等）を除去する。Blob 保存名は GUID を使用し、元ファイル名は DB テーブルで管理する。 |

---

### 8.2.5. AnyForm OCR ファイル取込

OCR 処理後の CSV ファイルを Azure Blob Storage の `ocr` コンテナに配置して、OCR 一覧画面から選択したファイルを申請書画面に反映する。  
※詳細は「OCR 取込方式」シートに記載する。

---

### 8.2.6. 財務会計システム連携ファイル取込

財務会計システムは IF 用ファイルを SFTP 経由で Azure Blob Storage の `finance` コンテナに配置する。入金実績画面にファイルを読み込む。  
※詳細は「財務連携取込方式」シートに記載する。

## 8.3. ファイルダウンロード方式

### 8.3.1. 帳票ダウンロード仕様

帳票（PDF / Excel / CSV）のダウンロードは、業務データを取得してサーバーサイドで MemoryStream に直接生成し、Blob Storage に保存することなく `FileContentResult` としてクライアントに直接返す。AP サーバーのローカルディスク・Blob Storage のいずれにも一時ファイルを作成しない。

Angular 側は HttpClient で `responseType: 'blob'` としてレスポンスを受け取り、file-saver ライブラリの `saveAs()` を使用してブラウザの保存ダイアログを起動する。

---

### 8.3.2. レスポンスヘッダ

各ダウンロードに対して以下のレスポンスヘッダを設定する。

**表8-5 ファイルダウンロードレスポンスヘッダ詳細**

| ヘッダ名 | 型 | 設定内容 |
|---|---|---|
| `Content-Disposition` | string | `attachment; filename*=UTF-8''{Uri.EscapeDataString(originalFileName)}`（RFC 5987 準拠）。日本語ファイル名に対応する。 |
| `Content-Type` | string | ファイル種別に応じた MIME タイプ。例：`application/pdf` / `text/csv; charset=utf-8` |
| `Content-Length` | long | `MemoryStream.Length` を設定する。ブラウザ側でダウンロード進捗が表示される。 |
| `Cache-Control` | string | `no-store, no-cache` を設定し、ブラウザキャッシュを無効化する。 |
| `X-Content-Type-Options` | string | `nosniff` を設定し MIME スニッフィングを防止する。 |

---

### 8.3.3. BOM（バイトオーダーマーク）の付加

CSV ファイルを UTF-8 で出力する際は、Microsoft Excel が文字コードを正しく判別できるよう、ファイル先頭にバイトオーダーマーク（`0xEF 0xBB 0xBF`）を付加する。  
.NET では `new System.Text.UTF8Encoding(encoderShouldEmitUTF8Identifier: true)` を使用する。

---

### 8.3.4. ダウンロード帳票種別

**表8-6 ダウンロード帳票種別**

| 帳票種別 | 出力ライブラリ | 備考 |
|---|---|---|
| PDF 帳票 | QuestPDF | MemoryStream に生成後、FileContentResult で直接返却する。 |
| Excel 帳票 | ClosedXML | MemoryStream に生成後、FileContentResult で直接返却する。 |
| CSV 帳票 | StreamWriter + StringBuilder | データ出力・外部連携用 CSV。BOM 付き UTF-8 で出力する。 |
