# 第１１章　異常処理方式

本章では、新たなシステム（以下「本システム」）において異常が発生した場合の処理方式について整理する。  
オンライン処理で異常が発生した場合、業務処理では異常の種別・対処方法に応じて適切なエラーハンドリングを実施し、統一されたレスポンス構造でフロントエンドに返却する。  
異常パターンごとのハンドリングについては基盤処理にて行う。

---

## 11.1 オンライン処理異常の分類

オンライン処理で発生する異常は、「処理継続可能のエラー」「処理継続不可のエラー」の二つに分類する。

### 11.1.1 処理継続可能のエラー

業務ロジックによる判定で発生したエラーが該当する。  
処理継続可能なエラーは、「画面にエラーメッセージ表示」の方法でユーザに通知する。  
共通例外処理クラスがメッセージコードを取得し、フロントエンドにエラーを返却する。  
フロントエンドは、レスポンスで受け取ったエラーの処理を行い、メッセージコード種別に応じて、該当するエラーメッセージを表示する。

- 「入力検証エラー」は、入力内容を修正することで処理継続可能なエラーが該当する。
- 「業務ロジックによる判定で発生したエラー」は、時間経過・再実行などで処理継続可能なエラーが該当する。

入力データが未入力またはデータ形式が正しくない場合、フロントエンドはリクエストをバックエンドへ送信する前に自動で入力データをチェックする。

- 画面の最上部に入力内容に誤りがある旨のエラーメッセージを表示する。
- 入力形式チェックのエラーメッセージは対応する入力欄の下にインラインで表示する。

### 11.1.2 処理継続不可のエラー

システム管理者や運用・保守担当者が原因を取り除かない限り回避不可のエラーが該当する。  
フロントエンドはエラー画面（専用画面）へ遷移する。ExceptionHandlerMiddleware が HTTP 500 を返却し、ErrorInterceptor が重度エラー専用画面（`/error/fatal`）へ誘導する。

### 表11-1 オンライン処理異常の分類詳細

| エラー種別 | 発生契機 | ユーザ通知方法 | 通知先 |
|---|---|---|---|
| 処理継続可能のエラー | バリデーション（未入力、桁数不正、入力形式不正、機種依存文字不正、範囲不正、日付不正）、業務チェックエラー、検索対象なし、選択データなし、複数件選択不可、削除不可、重複チェック、認証コード有効期限切れ等 | エラーメッセージ表示 | 利用者 |
| 処理継続不可のエラー | マスタ不整合、プロパティファイル設定エラー、ログイン処理異常（ユーザ存在なし、ロック回数更新失敗）、アクセス処理失敗、Entra ID 接続エラー、画面遷移不正、セッションタイムアウト（Entra ID トークン有効期限切れ）、排他制御、参加資格無効、システム利用停止、利用期間外等 | 重度エラー画面表示 | 運用・保守担当者 |

## 11.2 異常パターンによるハンドリング

オンライン処理で何らかの異常を発生させる場合、状況に応じてアプリケーションの実装方式を使い分けて実現する。

### (1) エラーメッセージを表示した状態で自画面に遷移させたい場合

基盤部品経由でエラー格納用のインスタンスにメッセージコードを設定する。

### (2) エラー画面に遷移させたい場合

例外（Exception）を発生させて、そのパラメータにメッセージコードを設定する。  
エラー画面（`/error/fatal`）へ遷移し、画面は重度エラー画面のみで構成される。

## 参考：実装詳細（Angular + .NET 10）

### バックエンド Middleware Pipeline

```
❶ ExceptionHandlerMiddleware  → 全例外キャッチ → errList[ESYS50001] + Serilog出力
❷ Serilog RequestLogging      → 全リクエスト/レスポンスをログに記録
❸ Authentication              → Microsoft.Identity.Web が Entra ID JWT を検証
❹ Authorization               → [Authorize(Roles=...)] / [Authorize(Policy=...)]
❺ MapControllers              → Controller にルーティング
```

### HTTPステータスコードとフロントエンド処理の対応

| HTTPステータス | エラー種別 | バックエンド処理 | フロントエンド処理 |
|---|---|---|---|
| 200 OK（errList あり） | 処理継続可能なエラー | ServiceResult → errList | ngx-toastr 赤表示 |
| 400 | 入力バリデーションエラー | FluentValidation → errList | インラインエラー表示 |
| 401 | 認証エラー | JWT 検証失敗 | MSAL 再ログインへリダイレクト |
| 403 | 認可エラー | Authorize 不適合 | アクセス拒否画面へ遷移 |
| 500 | 処理継続不可（重度） | ExceptionHandlerMiddleware | 重度エラー画面（`/error/fatal`）へ遷移 |

### メッセージコード体系

```
{レベル}{サブシステム}{種別コード5桁}

レベル      : I（情報 / nrmList）  W（警告 / wrnList）  E（エラー / errList）
サブシステム: HOSHO / SATEI / SUISHIN / KEIRI / KYOTSU / KASHI / KITAKU / SYS
種別コード  : 2xxxx（情報）  3xxxx（警告）  4xxxx（エラー）  5xxxx（システム）

例: IHOSHO21001 → 情報 / HOSHO / 21001番
例: WHOSHO31001 → 警告 / HOSHO / 31001番
例: EHOSHO41001 → エラー / HOSHO / 41001番
例: ESYS50001   → システムエラー（全サブシステム共通）
```

### メッセージコード例

| メッセージコード | 種別 | メッセージ内容（例） |
|---|---|---|
| IHOSHO21001 | 情報（nrmList） | 検索結果が0件です。 |
| WHOSHO31001 | 警告（wrnList） | 保証期間の終了日が近づいています。 |
| EHOSHO41001 | エラー（errList） | 入力された保証番号が存在しません。 |
| ESATEI41001 | エラー（errList） | 削除対象のレコードが他ユーザにより更新されています。 |
| ESYS50001 | 重度エラー（errList） | システムエラーが発生しました。システム管理者にお問い合わせください。 |
