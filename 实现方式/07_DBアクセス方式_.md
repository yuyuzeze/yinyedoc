# 第 7 章　DBアクセス方式

本章では、ホシステムにおけるDBアクセス方式について整理する。

本システムはデータベースとして Azure SQL Server を採用しており、Entity Framework Core 10 を中心にデータアクセスを実現する。

---

## 7.1. 処理方式

業務処理にてDBに対し何らかの操作を必要とする場合は、以下に示すDBアクセス部品を利用することで処理を実現する。

本システムは EF Core 10 の `Repository<T>` / `UnitOfWork` パターンを基盤が提供する共通部品として採用し、全サブシステムにて統一的に利用する。

### 表7-1　DBアクセス部品詳細

| 部品名 | 処理内容 | 処理詳細 | パラメータ | 戻り値 |
|--------|---------|---------|-----------|--------|
| `GetByIdAsync` | 主キー検索 | 指定されたIDに一致するエンティティを1件返却する。存在しない場合は `null` を返す。 | 主キー値 (TKey) | エンティティ or null |
| `GetAllAsync` | 全件取得 | テーブルの全レコードを非同期で取得し `IEnumerable<T>` で返却する。 | なし | `IEnumerable<T>` |
| `FindAsync` | 条件検索 | 検索条件をもとに検索用SQLを発行し、条件に一致するエンティティ一覧を返却する。 | 検索条件式 (Expression) | `IEnumerable<T>` |
| `FindPagedAsync` | 条件検索（ページング） | 検索条件に加え、ページ番号・ページサイズを指定して該当範囲のレコードのみを返却する。 | 検索条件式 / ページ番号 / ページサイズ | `PagedResult<T>` |
| `AddAsync` | 登録 | 命令情報に設定した登録値をもとに登録用SQLを発行する。 | エンティティ (T) | T（追跡中） |
| `Update` | 更新 | 命令情報に設定した検索条件・更新値をもとに更新用SQLを発行する。 | エンティティ (T) | void |
| `Remove` | 削除 | 命令情報に設定した検索条件をもとに削除用SQLを発行する。 | エンティティ (T) | void |
| `SaveChangesAsync`（UnitOfWork） | コミット | DBContext 上の全変更（登録/更新/削除）をトランザクション内でAzure SQLに一括送信する。 | なし | int（影響件数） |
| `QueryAsync`（Dapper） | 複雑クエリ | EF Core では表現困難な複雑な集計・結合クエリをDapper経由で直接SQLを発行し結果を返却する。 | SQL文字列 / パラメータ | `IEnumerable<T>` |

### 表7-2　DBアクセス時にやり取りされる情報詳細

| 情報区分 | プロパティ | 型 | 説明 |
|---------|-----------|-----|------|
| 検索条件 | `predicate` | `Expression<Func<T,bool>>` | LINQ式でWHERE句を構築する |
| | `pageNumber` | `int` | ページ番号（1始まり） |
| | `pageSize` | `int` | 1ページあたりの件数 |
| `PagedResult<T>` | `Items` | `IEnumerable<T>` | 該当ページのレコード一覧 |
| | `TotalCount` | `int` | 検索条件に一致するレコードの総件数 |
| | `PageNumber` | `int` | 現在のページ番号 |
| | `PageSize` | `int` | ページサイズ |
| | `HasNextPage` | `bool` | 次ページが存在するかを示すフラグ |

※ 結果格納用PagedResultの設定値は機能の処理内容に応じて異なる。

### 表7-3　処理内容とDBアクセス部品の対応マトリクス

| | GetAll Async | Find Async | Find Paged Async | Add Async | Update | Remove |
|--|:--:|:--:|:--:|:--:|:--:|:--:|
| 処理結果件数 | ○ | ○ | ○ | ○ | ○ | ○ |
| 検索結果データ | ○ | ○ | ○ | － | － | － |
| 検索結果総件数 | △※ | △※ | ○ | － | － | － |
| 次ページフラグ | － | － | ○ | － | － | － |

※ `GetAllAsync` / `FindAsync` は総件数を返さない。ページング画面では `FindPagedAsync` を使用すること。

## 7.2. DB接続方式

本システムは Azure VM の Managed Identity を使用してAzure SQL Database接続する。

VMに割り当てられたIDでEntra IDがトークンを発行し、そのトークンで接続する。接続文字列にパスワードが含まれないため、Key VaultによるDB接続文字列のシークレット管理も不要である。

**appsettings.json（接続文字列 — パスワードなし・Key Vault参照不要）**

```json
{
  "ConnectionStrings": {
    "AppDB": "Server=tcp:appdb.database.windows.net,1433;Initial Catalog=AppDB;Encrypt=True;TrustServerCertificate=False;Connection Timeout=30;Authentication=Active Directory Managed Identity;"
  }
}
```

そのうち、AppDB次期林業業務システムの業務データベース（Azure SQL Database）への接続文字列を指定する。各環境ごとに `appsettings.{Environment}.json` で接続先を切り替える。

## 7.3. トランザクション制御

EF Core 10 の DbContext および UnitOfWork パターンを拡張したトランザクション制御部品を基盤として提供する。この部品を業務処理で利用することでコネクションの取得、正常時のコミット、例外発生時のロールバックの制御を行う。

業務処理でトランザクション制御を入れることは可能だが、原則的にSQL単位ではコミットは行わない。`IUnitOfWork.SaveChangesAsync()` を一度呼び出すことで、当該リクエスト内の全変更を一括コミットする。

---

## 7.4. 排他制御

共通・基盤処理での排他制御は行わないため、制御を必要とする処理については各業務処理にて実装する。

トランザクション間の排他は、RowVersionを使用した楽観的排他制御方式で実装する（図7-2参照）。

トランザクション内の排他は、行排他（`SELECT ... WITH(UPDLOCK)`）を使用して対応する。

### 図7-2　楽観的排他制御方式イメージ

```
ユーザーA
  ① データ取得   [ID=1, 値=A, RowVersion=0x001]
  ③ 値の更新     [ID=1, 値=B]
  ⑤ 更新実行 → RowVersionがデータ取得時と変わらないので更新可
             → WHERE RowVersion=0x001 → 一致 → UPDATE成功 (RowVersion=0x002)

DB
  ID │ 値 │ RowVersion
   1 │ A  │ 0x001      ──────────────────────────→  ID=1, B, 0x002

ユーザーB（並行操作）
  ② データ取得   [ID=1, 値=A, RowVersion=0x001]
  ④ 値の更新     [ID=1, 値=C]
  ⑥ 更新実行 → RowVersionがデータ取得時と変わっているので更新不可
             → WHERE RowVersion=0x001 → 不一致（既に0x002）
             → DbUpdateConcurrencyException
             → 検索結果などの初期画面に戻り、更新できなかった旨のメッセージを表示
```

## 7.5. 暗号化方式

静的データの暗号化（TDE）は Azure SQL Database が自動的に行うため、アプリケーション層での追加対応は不要である。

## 7.6. デッドロック制御

Azure SQL Database でデッドロックが発生した場合、EF Core の `EnableRetryOnFailure` によりリトライを最大3回実施する。

リトライ上限を超えた場合は例外をキャッチし、`ExceptionHandlerMiddleware` 経由で errList にシステムエラーを格納してクライアントに返却する。

業務処理における更新順序を統一し、デッドロックの発生を予防的に抑止する。

### 表7-4　デッドロック発生時のSQLリトライ設定

| | オンライン | 非同期期（BackgroundService） |
|--|:--:|:--:|
| SQLタイムアウト | 85秒 | 300秒 |
| リトライ回数 | 3回 | 3回 |
| リトライ間隔 | 指数バックオフ（EF Core 既定） | 指数バックオフ（EF Core 既定） |

※ 定義値は `appsettings.json` の DbSettings セクション（Key Vault 参照）に定義する。
